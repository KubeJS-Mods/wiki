# AI Assistant Context for KubeJS Wiki

## Project Goal

This repository contains documentation for **KubeJS**, a Minecraft mod that allows players to write JavaScript to customize their gameplay. The wiki has **many missing documentation pages** for events, methods, and features that exist in the source code but are not yet documented.

**Primary Objective:** Fill in missing documentation by referencing the KubeJS source code and writing high-quality docs that match the existing style.

## How to Help

### 1. Access the Source Code

Use the `github_repo` tool (if available) to read the contents of the KubeJS source repository:

- **Repository:** `https://github.com/KubeJS-Mods/KubeJS`
- **Owner:** `KubeJS-Mods`
- **Repo Name:** `KubeJS`

This source code contains:

- Java files with `@Info` annotations explaining what methods do
- Event handler classes that define available events
- Examples and parameter descriptions

### 2. Understand the Wiki Structure

This wiki uses:

- **`.kubedoc` files** - Custom markup format (similar to Markdown but with special syntax)
- **`.yml` files** - Localization/translation files for different languages (en, de, pl, zh-cn, zh-tw)
- **Multilingual support** - All content should eventually be translatable

Key directories:

- `wiki/events/` - Event documentation (PlayerEvents, ServerEvents, etc.)
- `wiki/tutorials/` - Step-by-step guides
- `wiki/concepts/` - Conceptual explanations
- `wiki/ref/` - API reference documentation

### 3. Match Existing Documentation Style

Before writing new docs, **always check existing documentation** to match:

- **Writing style** - Clear, beginner-friendly, with examples
- **Code examples** - Practical, real-world use cases
- **Structure** - Consistent heading levels, parameter explanations
- **Format** - Proper `.kubedoc` syntax

Good examples to reference:

- `wiki/events/ServerEvents/` - Well-documented event examples
- `wiki/tutorials/` - Good tutorial structure
- `wiki/concepts/` - Well-explained conceptual topics

### 4. Writing Process

When documenting a missing feature:

1. **Find the source code** in the KubeJS repository
2. **Read existing similar docs** to understand the style
3. **Extract information** from:
   - `@Info` annotations in Java code
   - Method signatures and parameters
   - Existing examples in the codebase
4. **Write documentation** that includes:
   - Clear explanation of what it does
   - When/why to use it
   - Code examples (practical scenarios)
   - All parameters explained
   - Common pitfalls or gotchas
5. **Create `.kubedoc` file** following the existing format

## Common Tasks

- **Document missing events** in `wiki/events/`
- **Add tutorial content** for common use cases
- **Explain complex concepts** that users struggle with
- **Provide code examples** for poorly documented features

## Important Notes

- Focus on **practical, usable documentation** over technical accuracy
- Examples should be **copy-paste ready** for Minecraft modpack developers
- Always check if documentation **already exists** before creating new files

## File Formats

### `.kubedoc` Format

Custom markup similar to Markdown. Check existing files for syntax examples.

### `.yml` Format

Used for translations and metadata. Each language has its own `.yml` file.

## Questions to Ask

When unclear about what to document:

1. Which specific event/feature needs documentation?
2. What's a good existing doc to use as a style reference?
3. Should this be a tutorial, reference doc, or concept explanation?
